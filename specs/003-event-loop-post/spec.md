## User Scenarios & Testing *(mandatory)*

### User Story 1 - 시뮬레이터로 비동기 실행 순서 이해 (Priority: P1)

백엔드 스터디 멤버(JS가 주 언어가 아닌 개발자)가 글을 읽으며, 본문에 삽입된 스텝 실행 시뮬레이터를 직접 조작한다. "다음" 버튼을 누를 때마다 코드의 현재 실행 줄이 하이라이트되고, 콜스택·Web APIs·마이크로태스크 큐·태스크 큐·출력이 그 순간의 상태로 갱신되며, 이 스텝에서 무슨 일이 일어났는지 한 줄 설명이 함께 표시된다. 글이 진행될수록 시뮬레이터에 패널이 하나씩 추가되어(콜스택만 → +Web APIs → +태스크 큐 → 풀 구성) 독자의 머릿속 모델과 화면이 같이 자란다.

**Why this priority**: 이 기능이 곧 이 글의 존재 이유. 글 텍스트만으로는 기존 자료와 차별화되지 않으며, "시각적으로 만져보면서" 이해하는 것이 사용자의 핵심 목표.

**Independent Test**: 시뮬레이터가 삽입된 글 페이지를 열어 예제 하나를 처음부터 끝까지 스텝 이동(다음/이전/처음부터)하면서 각 패널이 스텝 데이터대로 갱신되는지 확인하면 단독 검증 가능.

**Acceptance Scenarios**:

1. **Given** 시뮬레이터가 삽입된 글 페이지, **When** 독자가 "다음"을 누르면, **Then** 다음 스텝의 스냅샷(하이라이트 줄·콜스택·큐·출력·설명)이 표시된다
2. **Given** 중간 스텝 상태, **When** "이전"을 누르면, **Then** 직전 스텝 상태가 완전히 복원된다 (비대칭 없음)
3. **Given** 마지막 스텝, **When** "다음"을 누르면, **Then** 더 진행되지 않고 마지막 상태가 유지된다 (버튼 비활성 등 명시적 표시)
4. **Given** 임의 스텝, **When** "처음부터"를 누르면, **Then** 첫 스텝으로 돌아간다
5. **Given** 패널 부분 표시로 삽입된 시뮬레이터(예: 콜스택만), **When** 렌더되면, **Then** 지정된 패널만 표시된다
6. **Given** 키보드 포커스가 시뮬레이터에 있는 상태, **When** ←/→ 키를 누르면, **Then** 이전/다음 스텝으로 이동한다

### User Story 2 - 서사를 따라 읽고 실무 연결·토론 거리 획득 (Priority: P2)

독자가 도입 퀴즈(setTimeout과 Promise가 섞인 코드의 출력 순서 맞히기)에서 먼저 틀리고 시작해, 콜스택 → Web API → 태스크 큐 → 마이크로태스크 큐 → 프로미스 순서로 개념을 쌓은 뒤, 글 끝에서 도입 퀴즈를 다시 풀며 "이제 읽을 수 있다"를 경험한다. 각 개념 섹션 끝에는 "실무에서는"(이 개념을 실제로 만나는 곳)과 "생각해볼 점"(함정·트레이드오프를 질문 형태로 — 스터디 토론 거리) 박스가 있고, Node.js 심화 내용은 접힌 부록으로 제공된다.

**Why this priority**: 사용자가 명시한 두 번째 축 — "어디서 뭘 사용하는지"와 "생각해봐야 할 점"이 중요. 다만 시뮬레이터(US1) 없이는 글이 성립하지 않으므로 P2.

**Independent Test**: 글을 처음부터 끝까지 읽으며 서사 구조(도입 퀴즈 → 개념 5단계 → 퀴즈 회수 → 마무리 퀴즈), 섹션별 Callout 2종, Node 심화 부록의 존재와 내용 정합을 확인하면 단독 검증 가능.

**Acceptance Scenarios**:

1. **Given** 게시된 글, **When** 목차를 확인하면, **Then** 도입 퀴즈 → 싱글스레드와 콜스택 → Web API → 태스크 큐와 이벤트 루프 → 마이크로태스크 큐 → 프로미스로 합치기 → 마무리 퀴즈 서사가 완결되어 있다
2. **Given** 각 개념 섹션, **When** 섹션 끝을 보면, **Then** "실무에서는"과 "생각해볼 점" 박스가 있다
3. **Given** 글 본문, **When** Node.js 심화(6페이즈, nextTick vs setImmediate, async/await 변환 상세)를 찾으면, **Then** 접힌 부록 형태로 제공되어 본문 흐름을 방해하지 않는다
4. **Given** 도입 퀴즈 섹션, **When** 독자가 보기 중 예측을 클릭해 제출하면, **Then** 맞음/틀림과 함께 시뮬레이터로 실행 순서가 공개되며 "지금은 이해 안 되는 게 정상"임이 명시된다 (제출 전에는 정답 미노출)

---

### User Story 3 - 시뮬레이션 정확성 자동 검증 (Priority: P3)

작성자(블로그 주인)가 손으로 작성한 스텝 스크립트가 실제 JS 동작과 다르면("시뮬레이션이 거짓말"), 자동 검증이 이를 잡아낸다. 각 예제의 코드는 실제 실행 가능하게 유지되고, 실제 실행 출력 순서와 시뮬레이션 최종 출력이 일치하는지, 스텝 데이터가 구조 규칙(콜스택 push/pop 정합, 출력 append-only, 하이라이트 줄 범위)을 지키는지 기계적으로 확인된다.

**Why this priority**: 손 작성 스텝 방식(채택한 접근법)의 최대 리스크에 대한 방어 장치. 독자 가치는 간접적이지만, 틀린 교육 자료는 없느니만 못하므로 게시 전 필수 게이트.

**Independent Test**: 예제 하나의 스텝 데이터를 의도적으로 틀리게 바꾼 뒤 검증이 실패하는지, 원복하면 통과하는지 확인하면 단독 검증 가능.

**Acceptance Scenarios**:

1. **Given** 모든 예제, **When** 자동 검증을 실행하면, **Then** 각 예제의 실제 실행 출력 순서와 시뮬레이션 마지막 스텝의 출력이 일치한다
2. **Given** 스텝 데이터에 구조 위반(예: 출력이 중간에 사라짐, 하이라이트 줄이 코드 범위 밖), **When** 검증을 실행하면, **Then** 해당 예제와 위반 내용이 명시된 실패가 보고된다
3. **Given** 게시된 글 페이지, **When** 자동 브라우저 테스트를 실행하면, **Then** 시뮬레이터가 렌더되고 다음/이전/처음부터 조작이 동작한다

---

### Edge Cases

- 존재하지 않는 예제 이름으로 시뮬레이터를 삽입하면? → 조용한 빈 화면이 아니라 명시적 에러 표시 (글 검증 단계에서도 잡히면 더 좋음)
- 첫 스텝에서 "이전", 마지막 스텝에서 "다음"을 누르면? → 진행되지 않으며 버튼 상태로 명시
- 좁은 화면(모바일)에서 패널 5개가 다 보여야 하면? → 세로 스택으로 재배치, 조작 가능 유지
- 독자가 움직임 최소화(prefers-reduced-motion)를 설정했다면? → 이동 애니메이션 없이 상태 전환만
- 다크모드에서 하이라이트·큐 색상이 보이는가? → 블로그 디자인 토큰만 사용해 자동 대응
- 스텝이 1개뿐인 예제? → 조작 버튼이 모두 비활성이어도 렌더는 정상

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 블로그에 "싱글스레드 JS의 비동기 처리(이벤트 루프·태스크 큐·마이크로태스크 큐·프로미스)"를 다루는 글 1편이 게시되어야 한다. 서사는 도입 퀴즈 → 싱글스레드와 콜스택 → 기다리는 동안 누가 일하나(Web API) → 태스크 큐와 이벤트 루프 → 마이크로태스크 큐 → 프로미스로 합치기(도입 퀴즈 회수) → 마무리 퀴즈 순서를 따른다
- **FR-002**: 독자는 글 안에서 스텝 실행 시뮬레이터를 조작할 수 있어야 한다 — "이전/다음/처음부터" 버튼과 키보드 ←/→, 각 스텝에서 현재 코드 줄 하이라이트·콜스택·Web APIs·마이크로태스크 큐·태스크 큐·누적 출력·한 줄 설명(note)을 표시
- **FR-003**: 시뮬레이터는 표시할 패널을 삽입 지점마다 다르게 지정할 수 있어야 한다 (점진 공개: 콜스택만 → 풀 구성)
- **FR-004**: 시뮬레이터 예제는 이름으로 참조하며, 존재하지 않는 이름이면 명시적 에러를 표시해야 한다 (조용한 실패 금지)
- **FR-005**: 예제는 6~7개를 제공해야 한다 — 최소 구성: 도입 퀴즈용, 콜스택 전용, 콜스택+Web API(setTimeout), 태스크 큐 합류, 마이크로태스크 새치기, 프로미스/async-await, 마무리 퀴즈용
- **FR-006**: 각 개념 섹션 끝에 "실무에서는"(사용처)과 "생각해볼 점"(함정·트레이드오프 질문형, 스터디 토론 거리) 박스 2종이 있어야 한다
- **FR-007**: Node.js 심화 내용(이벤트 루프 6페이즈, process.nextTick vs setImmediate, async/await 변환 상세)은 접힌 부록으로 제공되어야 한다
- **FR-008**: 시뮬레이터는 모바일 화면에서 조작 가능해야 하고, 움직임 최소화 설정을 존중해야 하며, 라이트/다크 테마 모두에서 가독성이 유지되어야 한다
- **FR-009**: 모든 예제에 대해 "실제 실행 출력 순서 = 시뮬레이션 최종 출력"을 자동 검증해야 한다 (예제 코드는 실제 실행 가능한 JS로 유지)
- **FR-010**: 스텝 데이터의 구조 규칙(콜스택 push/pop 정합, 출력 append-only, 하이라이트 줄이 코드 범위 내)을 자동 검증해야 한다. 스텝 굵기는 혼합이므로("한 스텝 = 변화 1종" 규칙은 채택하지 않음) 위 3가지가 무결성 검증의 전부이며, 정확성의 핵심 방어선은 FR-009의 출력 순서 대조다 (grilling Q4, 2026-07-23)
- **FR-011**: 게시된 글에서 시뮬레이터 렌더와 기본 조작(다음/이전/처음부터)이 동작하는지 자동 브라우저 테스트로 확인해야 한다
- **FR-012**: 도입·마무리 퀴즈는 선택형이어야 한다 — 독자가 보기(출력 순서 후보) 중 하나를 클릭해 예측을 제출하면 정답과 맞음/틀림이 공개된다. 제출 전에는 정답이 노출되지 않는다 (별도 퀴즈 요소 1종 추가, 2026-07-23 사용자 확정)
- **FR-013**: 퀴즈와 정답 공개용 시뮬레이터는 결합되어야 한다 — 예측을 제출하기 전에는 해당 시뮬레이터가 잠겨(또는 숨겨져) 있어 출력 패널로 정답을 미리 볼 수 없다 (grilling Q1, 2026-07-23)
- **FR-014**: `await`는 정확하게 표현해야 한다 — async 함수가 await에 도달하는 스텝에서 함수 프레임이 콜스택에서 내려가고, 프로미스 해결 시 "함수 이어서 실행" 항목이 마이크로태스크 큐를 거쳐 콜스택으로 복귀하는 것으로 표시한다. 프레임을 콜스택에 "일시정지"로 남겨두는 단순화는 금지 (grilling Q2, 2026-07-23)

### Key Entities *(include if feature involves data)*

- **예제 (SimExample)**: 시뮬레이터가 재생하는 하나의 시나리오. 이름(글에서 참조하는 식별자), 제목, 표시용 코드(실제 실행 가능한 JS, 줄 단위), 스텝 목록을 가진다
- **스텝 (SimStep)**: 특정 순간의 완전한 스냅샷. 하이라이트 줄, 콜스택, Web APIs(대기 중 작업), 마이크로태스크 큐, 태스크 큐, 누적 출력, 한 줄 설명(note)을 가진다. 스텝은 이전 스텝에 의존하지 않으므로 앞뒤 이동이 대칭이다. 굵기는 혼합: 동기 코드는 줄 단위로 빠르게, 큐 이동·await 분리·이벤트 루프 틱 같은 중요한 순간은 사건 단위로 세분화한다 (grilling Q4)
- **퀴즈 (Quiz)**: 출력 순서 보기 목록, 정답, 결합된 시뮬레이터 예제 참조를 가진다. 제출 전에는 시뮬레이터가 잠김 상태다 (grilling Q1)

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 독자는 글 안의 어떤 시뮬레이터든 처음부터 마지막 스텝까지 앞뒤로 자유롭게 탐색할 수 있다 (모든 예제, 데스크톱·모바일)
- **SC-002**: 시뮬레이션 최종 출력과 실제 실행 출력의 일치율 100% — 자동 검증이 게시 전 게이트로 동작한다
- **SC-003**: 글의 서사가 완결된다: 도입 퀴즈에서 제기한 질문이 글 끝에서 같은 예제로 회수되고, 모든 개념 섹션에 Callout 2종이 존재한다
- **SC-004**: 스터디 모임에서 "생각해볼 점" 질문만으로 토론을 진행할 수 있다 — 각 질문은 본문만 읽으면 논거를 댈 수 있도록 근거가 본문에 존재한다
- **SC-005**: 라이트/다크 테마, 모바일/데스크톱 4개 조합 모두에서 시뮬레이터 패널·하이라이트가 읽을 수 있는 상태다

## Assumptions

- 본문 개념 범위는 브라우저/Node 공통 원리(콜스택·태스크 큐·마이크로태스크 큐·프로미스) 기준이며, 런타임별 차이는 심화 부록에서만 다룬다 (설계 문서 결정)
- 예제 코드는 결정적 API만 사용한다: `console.log`, `setTimeout`, `Promise.resolve/then/catch`, `async/await`, `queueMicrotask` (grilling Q3). fetch 등 비결정적 API는 본문 설명과 "실무에서는" Callout에서만 다루고 예제 코드에는 넣지 않는다 — 출력 순서 자동 대조(FR-009)에 예외를 만들지 않기 위함
- 스텝 데이터는 손으로 작성한다 — 코드에서 스텝을 자동 생성하는 인터프리터는 범위 외 (설계 문서에서 기각)
- 독자 코드 입력(자유 시뮬레이션), 손그림 스타일 일러스트, 시리즈 분할은 범위 외
- 글은 한국어로 작성하며, 기존 블로그의 게시 파이프라인(등록된 컴포넌트만 글에서 사용 가능)을 따른다
- 시각 스타일은 기존 블로그 디자인 토큰을 재사용한다 (새 팔레트 도입 없음)
- 이 피처의 완료 = 글 공개까지. 머지 즉시 글이 공개되며, drafts 단계를 두지 않는다 (grilling Q5)
- 구현 프로세스 노트: 스텝 스크립트 중 핵심 2개(마이크로태스크 새치기, async/await 분리)와 글 본문 초안은 작성자가 직접 작성한다 — 학습 목표(비동기 이해)와 정확히 겹치는 부분. 형식·첫 예제·검증 테스트는 선행 제공되어 채점자 역할을 한다. 나머지 컴포넌트·테스트·인프라는 평소 프로세스대로 위임 (grilling Q6, tasks 단계 레인 설계에 반영)
