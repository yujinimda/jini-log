# 지니로그 (개인 블로그) 설계

- 날짜: 2026-07-21
- 상태: 사용자 승인 완료 (brainstorming 산출물)
- 범위: 블로그 MVP. AI 에이전트 자동 글감 캡처(원 기획 3번)는 별도 프로젝트로 후순위. 단, 초안 주입 입구는 이번 설계에 포함.

## 요구사항

아래 번호는 이 문서 기준 재번호 (원 기획의 1, 2, 4, 5, 6에 해당).

1. 블로그에서 직접 글을 작성할 수 있어야 한다
2. 검색에 잘 걸려야 한다 (SEO)
3. 글이 인터랙티브해야 한다 — 독자가 만지면서 이해
4. 마크다운으로 작성 가능해야 한다
5. 운영자용 대시보드가 있어야 한다

## 핵심 결정

| 결정 | 선택 | 이유 |
| --- | --- | --- |
| 콘텐츠 저장 | 파일 MDX + git 커밋 | 인터랙티브 컴포넌트 자유도 최대, 글 = 코드로 버전관리, 에이전트 초안 주입 입구가 공짜로 생김 |
| 에디터 | 마크다운 에디터 + 라이브 프리뷰 | 마크다운 익숙한 사용자에게 충분, MVP 난이도 적절. 위지윅은 이후 후보 |
| 발행/초안 구분 | 폴더 위치 (posts/ vs drafts/) | 가장 단순한 규약, 발행 행위가 git 이력에 명시적으로 남음 |
| 조회수/통계 | 자체 집계 (Supabase) | 대시보드 표현 자유도, 이후 인기글 정렬 등 확장 가능 |
| 어드민 인증 | GitHub OAuth, 허용 계정 1개 | GitHub API 커밋과 결이 맞고 비밀번호 관리 불필요 |
| 아키텍처 | 단일 Next.js 앱 | 레포·배포·스택 단일, 오르카 레인 분배 용이 |
| 배포 | Vercel | git 커밋 → 자동 재배포 흐름과 자연스럽게 결합 |

## 아키텍처

스택: Next.js(App Router) + TypeScript + Tailwind + Supabase(조회수 전용). 단일 레포, Vercel 배포.

```
app/
  (blog)/          # 공개 영역: 홈, 글 목록, 글 상세, 태그
  admin/           # 인증 필요: 에디터, 대시보드
  api/             # 조회수 기록, GitHub 커밋, 인증
content/
  posts/*.mdx      # 발행 글 (파일명 = slug)
  drafts/*.mdx     # 초안 — 이후 에이전트가 초안을 커밋할 입구
components/
  mdx/             # 글에서 쓰는 인터랙티브 컴포넌트 레지스트리
```

## 콘텐츠 모델

- 글 하나 = MDX 파일 하나. 파일명이 slug.
- frontmatter: `title`, `description`(SEO 요약), `date`, `tags`.
- 발행/초안은 frontmatter 플래그가 아니라 폴더 위치로 구분. 발행 = drafts/ → posts/ 파일 이동 커밋.
- 빌드 시 frontmatter를 파싱·검증해 타입 안전한 글 목록 생성 (도구 선정은 plan 단계 — velite 류 또는 자체 로더).

## 어드민

### 인증
- Auth.js(NextAuth) GitHub OAuth. 허용 GitHub 계정을 환경변수로 지정.
- `/admin` 전체와 쓰기 API를 미들웨어에서 차단.

### 에디터 (`/admin/write`)
- 좌: 마크다운 입력(CodeMirror 계열) / 우: 라이브 프리뷰.
- 프리뷰는 블로그 본문과 같은 MDX 컴포넌트 레지스트리로 렌더 — 인터랙티브 컴포넌트도 프리뷰에서 실제 동작.
- frontmatter는 폼 필드로 입력, 저장 시 합성 (오타로 인한 빌드 실패 원천 차단).
- 작성 중 내용은 localStorage 자동 백업 — 커밋 실패에도 유실 없음.

### 발행 플로우
- "초안 저장" → GitHub API로 `content/drafts/{slug}.mdx` 커밋.
- "발행" → `content/posts/{slug}.mdx` 커밋(초안이면 이동) → Vercel 자동 재배포.
- 수용한 트레이드오프: 발행 후 실제 반영까지 재배포 시간 1~2분.
- 기존 글 수정: GitHub API에서 최신 파일을 읽어와 편집 — 항상 최신본 기준.

### 대시보드 (`/admin`)
- 글 목록: 발행글 + 초안, 글별 조회수, 최근 발행일.
- 조회수: 글 페이지 비콘 → `/api/views` → Supabase에 slug·일자 단위 누적. 글별/기간별 추이 표시.
- 조회수 비콘에서 어드민 세션은 카운트 제외 — 개발·프리뷰 중 본인 조회로 숫자가 오염되는 것 방지. MVP에서 유일하게 처리하는 어뷰징 케이스.
- drafts/ 목록이 글감 인박스 역할 — 이후 에이전트가 초안을 커밋하면 자동으로 노출.

## SEO

- 모든 글·목록·태그 페이지 SSG.
- `generateMetadata`로 글별 title/description/OG 태그.
- OG 이미지 자동 생성 (`ImageResponse`, 글 제목 기반) — 글마다 썸네일 제작 불필요.
- `sitemap.xml` · `robots.txt` · RSS 피드 · JSON-LD(Article) 자동 생성.

## 인터랙티브 MDX

- `components/mdx/`가 레지스트리. 글에서 `<ErrorSimulator />`처럼 태그로 사용.
- 기본 제공: 코드블록(하이라이트 + 복사 버튼), Callout, 접기/펼치기.
- 경계: 새 인터랙티브 컴포넌트는 로컬 개발로만 추가, 어드민 에디터에서는 사용만 가능. 워크플로우 = "데모 필요 → 로컬에서 컴포넌트 추가 → 글에서 태그 사용".

## 에러 처리

- 저장/발행 API가 커밋 전에 MDX 컴파일을 시도, 실패 시 커밋 거부 + 에러 위치 표시. 깨진 글이 레포에 들어가 빌드를 깨뜨리는 것 원천 차단.
- 조회수 기록은 fire-and-forget — 실패해도 독자 경험에 영향 없음.
- GitHub API 실패 시 에러 표시, localStorage 백업으로 내용 보존.

## 테스트

- 유닛: 콘텐츠 로더 · frontmatter 검증 · MDX 컴파일 검증.
- API: 조회수 기록 · 커밋 플로우 (GitHub API 모킹).
- E2E: "작성 → 초안 저장 → 발행 → 공개 페이지 노출" 데모 시나리오. 개발 프로세스 8단계(머지 후 데모 검증)에서 그대로 사용.
